## 🏗 Архитектура системы (In Active Development)

Проект находится в процессе миграции от стабильного монолитного скрапера к отказоустойчивой распределенной архитектуре. Система разделяется на независимые изолированные слои (микросервисы):

1. **Скрапер Cluster (`cmd/scraper`)** — *[Текущий этап]* Фоновые воркеры. Управляют пулом безголовых браузеров (`chromedp`). В процессе перевода с локальных Go-каналов (`chan Job`) на атомарный забор задач из распределенной очереди Redis. Отправляют сырые данные (JSON) напрямую в Kafka.
2. **Брокер данных (Kafka)** — *[Инфраструктура готова]* Выступает в качестве отказоустойчивого буфера. Позволяет полностью разорвать жесткую связь между скрапером и базой данных, предотвращая пиковые нагрузки на СУБД.
3. **API Backend / Consumer (`cmd/api`)** — *[В разработке]* Веб-сервер и консьюмер. В фоновом режиме слушает топики Kafka, валидирует данные, сохраняет их в PostgreSQL и обрабатывает входящие HTTP-запросы от пользователей.

### Стек технологий
* **Language:** Go (Golang)
* **Databases:** PostgreSQL (основное хранилище), Redis (распределенная очередь задач, кэширование API)
* **Message Broker:** Apache Kafka (KRaft mode)
* **Scraping:** Chromedp (Headless Chrome)
* **Migration:** Golang-Migrate
* **Containerization:** Docker, Docker Compose

## 🚀 Ключевые архитектурные паттерны (Проектируемые и Внедряемые)

* **Distributed Task Queue (Redis BLPop/LPush):** *[В процессе интеграции]* Задачи на парсинг категорий переносятся в Redis. Это позволит запускать несколько независимых инстансов скрапера параллельно на разных машинах — Redis гарантирует атомарную выдачу задачи строго одному воркеру.
* **Event-Driven Data Ingestion:** *[В процессе интеграции]* Скрапер перестает зависеть от скорости дискового ввода-вывода PostgreSQL. Он отправляет событие с сырым JSON в Kafka за <1мс и сразу переходит к следующей задаче.
* **Cache-Aside Pattern (Redis Cache):** *[Проектируется]* Тяжелые аналитические SQL-запросы (расчет средних цен, позиций брендов) кэшируются в Redis для разгрузки PostgreSQL.
* **Graceful Shutdown:** *[Реализовано]* Все компоненты системы (включая БД-коннекты и контексты chromedp) корректно завершают работу при получении сигналов `SIGINT/SIGTERM` без потери данных.
